CI/CD jenkins 

Developed in Github -> Jenkins -> UE4 UAT -> Notify Tool | Groovy, Batch

Upon completing the development of Gloom Reaper, my curiosity was piqued by the potential for automation through Jenkins. To deepen my expertise in this domain, I started a personal project to establish my own Jenkins server at home, integrating automated testing into the pipeline. This endeavor proved challenging due to the scarcity of documentation about the UE ​Automation System. ​I had to delve into the engine source code to effectively implement the desired functionalities.

More information about this project.

I decided to utilize GitHub for its accessibility compared to Perforce. The project aimed to automate testing for the game "Gloom Reaper", leveraging my familiarity with the game to implement propper QA tests.

I had a clear vision for my Jenkins pipeline. The primary objectives were to establish a network of interconnected nodes, introduce task parallelism across nodes, and integrate automated tests into the pipeline to enhance the game's quality assurance process.

Setting up multiple nodes and PCs on the Jenkins server was a straightforward process. To mitigate race conditions inherent in parallel programming, each node operated on a local-updated copy of the project and generated its own result file. Upon task completion, a central machine combined all result files to create a consolidated output, which was then parsed.

Developing automated tests proved more intricate due to the lack of documentation. I found several problems
during the testing phase:

  • Unit Tests: Information on UE unit testing was scarce, prompting me to search into the predefined UE source code tests to discern the appropriate macros, classes, and functions that i should use for my tests.
  • Functional Tests: While functional tests had more documentation, they presented their own set of challenges. Incorporating additional test log information, such as identifying live enemies during a combat test failure, required utilizing a function provided by UE's functional tests class. However, this function was commented out in the engine code, necessitating a workaround to fulfill the required functionality.
  • Execution Command Issues: The command-line execution of tests also had its own share of dificulties. Jenkins, primarily in the Windows environment, lacked adequate privileges to access the PC GPU for a visual validation of tests. Although non-UI execution was possible, DirectX-related crashes and plugin issues, such as FMOD returning null values, posed obstacles.

I managed to overcome most of these challenges, while gaining experience in the process.
