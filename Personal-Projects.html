CI/CD jenkins 

Developed in Github -> Jenkins -> UE4 UAT -> Notify Tool | Groovy, Batch

Upon completing the development of Gloom Reaper, my curiosity was piqued by the potential for automation through Jenkins. ​To deepen my expertise in this domain, I initiated a personal project to set up my own Jenkins server at home. This involved integrating automated testing into the pipeline, parallelizing tasks, and utilizing different nodes. This endeavor proved challenging due to the scarcity of documentation about the UE ​Automation System. ​I had to delve into the engine source code to effectively implement the desired functionalities.

More information about this project.

I decided to utilize GitHub for its accessibility compared to Perforce. The project aimed to automate testing for the game "Gloom Reaper", leveraging my familiarity with the game to implement propper QA tests.

I had a clear vision for my Jenkins pipeline. The primary objectives were to establish a network of interconnected nodes, introduce task parallelism across nodes, and integrate automated tests into the pipeline to enhance the game's quality assurance process.

Setting up multiple nodes and PCs on the Jenkins server was a straightforward process. To mitigate race conditions inherent in parallel programming, each node operated on a local-updated copy of the project and generated its own result file. Upon task completion, a central machine combined all result files to create a consolidated output, which was then parsed.

Developing automated tests proved more intricate due to the lack of documentation. I found several problems
during the testing phase:

  • Unit Tests: Information on UE unit testing was scarce, prompting me to search into the predefined UE source code tests to discern the appropriate macros, classes, and functions that i should use for my tests.
  • Functional Tests: While functional tests had more documentation, they presented their own set of challenges. Incorporating additional test log information, such as identifying live enemies during a combat test failure, required utilizing a function provided by UE's functional tests class. However, this function was commented out in the engine code, necessitating a workaround to fulfill the required functionality.
  • Execution Command Issues: The command-line execution of tests also had its own share of dificulties. Jenkins, primarily in the Windows environment, lacked adequate privileges to access the PC GPU for a visual validation of tests. Although non-UI execution was possible, DirectX-related crashes and plugin issues, such as FMOD returning null values, posed obstacles.

I managed to overcome most of these challenges, while gaining experience in the process.

Improved AI

Developed in UE 4.27 | C++, BPs

While learning in the master's degree, we learned about different AI techniques that I coudnt try while Gloom Reaper's development, in order to practise and gain knowledge I tried to improve different AIs.
- I wanted to test out navLinks and see how they work. To do that I created a player AI capable of finishing the game using lots of different navLinks to be able to complete the platforming sections. This navLinks imitate how a player would play, for example, how many ms should the space mantained to balance a jump ​height. 
- In order to create a more organized combat, y created a combat manager that steers the different active enemies. From all enemies in a combat, a few of the enemies will try to get to the player, and only a few of them will actualy attack while the rest of the enemies are waiting outside of player range for their combat turn. After a few attacks of a single enemy, it will go to the waiting group while a waiting enemy will join the battle.
- Flocking is an interesting technique but none of the enemies really fits the technique. I sort of adapted the separation rule to the rabbits  to make their movement more fluid while evading themselves and having a target.

More information about my work in this project.

The AI player exhibits two distinct behaviors: a combat behavior (relatively simple) and a movement behavior that aims to find a path to the end of the level and traverse it.
To establish this path, I added various navLinks throughout the game to connect different sections. Some of these links prompted the AI to jump, others compelled the AI to execute attacks or employ skills, while some combined functionalities such as jumping followed by a dash after a certain duration.
During this process, I recognized the utility of automatically adding navLinks to each obstacle. This could be accomplished either by incorporating them into the obstacle blueprint or by devising an automated tool to directly add them to the level. Otherwise every time someone of the design team sightly changes an obstacle in a level, the navLink would broke.

The concept behind the combat manager entails an object featuring an 'in-combat' array housing enemies attempting to reach the player, an 'idle' array encompassing enemies in a waiting state, and thresholds for maximum in-combat and attacking points. These thresholds were implemented to dynamically limit the number of enemies. Smaller or easier enemies possessed lower weights, enabling more of them to coexist, while larger or more challenging enemies carried greater weights and often engaged in smaller groups.

The combat manager posed challenges; initially, I attempted to adapt it to the existing enemies. However, lots of fixes and updates were necessary, rendering it unmanageable. Subsequently, I opted to reconstruct both enemy types from the ground up, considering the combat manager's framework. Even then, I unintentionally tailored the combat manager overly specifically for worms when redeveloping the system, making the integration of other enemies a challenge.


While understanding the fundamentals of flocking is relatively straightforward, applying them to rabbit behavior proved intricate due to their jumping movement pattern. A recurring issue happened when one rabbit initiated a jump, and another rabbit moved in its path, causing the former to abruptly alter its course mid-air. This swift alteration didn't align with the expected behavior of a rabbit. To address this, I introduced adjustments like implementing a maximum rotational speed and applying flocking exclusively during the initial phase of the jump, resulting in a more natural appearance. However, a drawback of this approach is occasional rabbit-to-rabbit collisions.
