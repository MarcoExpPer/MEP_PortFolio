Gloom Reaper

Developed in UE 4.27 | C++

I contributed to the development of a sidescroller hack and slash game as part of a large team consisting of over 30 individuals. My primary focus was on key aspects of the gameplay, including the player movement and attack controller, the creation of the rabbit enemy, the implementation of combat zones, and the inclusion of various destructible elements throughout the game. 

More information about my work in this project.

This project was the culmination of my master's degree program, involving collaboration between various departments and teams, such as art, production, and design. Despite the large number of people involved, only a select few were actively contributing to the project. As a result, I took on a leadership role within the programming team, closely collaborating with the design department and supporting the production team in organizing and allocating programming tasks to individuals with the most involvement and expertise in the project.

As a programmer, my primary contributions revolved around several key aspects of the game. I developed the player controller,  implemented the behavior of the rabbit enemy, created the combat zones, and programmed the destructible elements. Additionally, I worked on various other components of the game, including traps, camera functionality, shaders, UI improvements, and the implementation of a comprehensive respawn system for all game elements.

I took the initiative to fully automate our game's compilation. By utilizing tools such as Jenkins and Perforce and setting up a Discord server, we were able to automatically compile and distribute the builds. This enabled us to quickly identify the source of any compilation errors.

Due to the lack of active contributors, I also took on the role of a level designer. I designed the majority of the game's levels, crafted the rabbit enemy's characteristics and behavior, and contributed to the design of combat mechanics. Furthermore, I collaborated with the production team to streamline the project by removing unnecessary mechanics, resulting in a reduced workload while ensuring a polished and cohesive gameplay experience.

The game is currently available to free download in steam: steam link.

CI/CD jenkins 

Developed in Github -> Jenkins -> UE4 UAT -> Notify Tool | Groovy, Batch

Upon completing the development of Gloom Reaper, my curiosity was piqued by the potential for automation through Jenkins. ​To deepen my expertise in this domain, I initiated a personal project to set up my own Jenkins server at home. This involved integrating automated testing into the pipeline, parallelizing tasks, and utilizing different nodes. This endeavor proved challenging due to the scarcity of documentation about the UE ​Automation System. ​I had to delve into the engine source code to effectively implement the desired functionalities.

More information about this project.

I decided to utilize GitHub for its accessibility compared to Perforce. The project aimed to automate testing for the game "Gloom Reaper", leveraging my familiarity with the game to implement propper QA tests.

I had a clear vision for my Jenkins pipeline. The primary objectives were to establish a network of interconnected nodes, introduce task parallelism across nodes, and integrate automated tests into the pipeline to enhance the game's quality assurance process.

Setting up multiple nodes and PCs on the Jenkins server was a straightforward process. To mitigate race conditions inherent in parallel programming, each node operated on a local-updated copy of the project and generated its own result file. Upon task completion, a central machine combined all result files to create a consolidated output, which was then parsed.

Developing automated tests proved more intricate due to the lack of documentation. I found several problems
during the testing phase:

  • Unit Tests: Information on UE unit testing was scarce, prompting me to search into the predefined UE source code tests to discern the appropriate macros, classes, and functions that i should use for my tests.
  • Functional Tests: While functional tests had more documentation, they presented their own set of challenges. Incorporating additional test log information, such as identifying live enemies during a combat test failure, required utilizing a function provided by UE's functional tests class. However, this function was commented out in the engine code, necessitating a workaround to fulfill the required functionality.
  • Execution Command Issues: The command-line execution of tests also had its own share of dificulties. Jenkins, primarily in the Windows environment, lacked adequate privileges to access the PC GPU for a visual validation of tests. Although non-UI execution was possible, DirectX-related crashes and plugin issues, such as FMOD returning null values, posed obstacles.

I managed to overcome most of these challenges, while gaining experience in the process.

Dungeon's Staff

Developed in UE 5.1.1 | Blueprints

This game is a ​rhythm-stealth minigame-based game I worked on for the GMTK jam as a part of a 10 people team. I handled all the minigames, the game rules, all aspects of the UI, and the interactive items that are integral to gameplay.

More information about my work in this project.

Our team participated in the GMTKGameJam, completing this project within a tight 48-hour timeframe. We worked together as a multidisciplinary team of 10 people, using the blueprints system for development. We chose this approach because we believed it would enable faster iteration and development compared to constantly compiling our games in C++. Additionally, using blueprints allowed non-programmers companions to contribute to the game's logic and programming.

As part of my contributions to the game, I focused on developing the minigames and designing the various interactable elements.
Two standout minigames were the 'press action when the big circle is aligned with the small one' and the 'imitate the initial beat' game. These minigames drew inspiration from mechanics found in Hi-Fi Rush. I take pride in these contributions because they required precise alignment of the UI and player inputs with the rhythm and beats of the songs and sound effects.

Aligning the music with the gameplay and UI posed a challenge for our team's musician and myself, but we managed to achieve a decent result with everything properly aligned. However, it is worth noting that despite the alignment being accurate and on beat, some of the minigames, particularly the "imitate the initial beat", may prove to be more challenging than expected due to the narrow window for correct inputs.

The game is currently available to download in itchi io: itch io link.
